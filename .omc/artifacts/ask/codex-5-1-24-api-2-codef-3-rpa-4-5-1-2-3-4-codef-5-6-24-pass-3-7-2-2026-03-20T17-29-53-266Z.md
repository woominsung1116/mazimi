1. **1단계: 반자동 수집**
- 사용자에게 정부24/홈택스 발급 링크 제공
- 간편인증으로 직접 발급하게 함
- PDF 업로드 또는 전자문서지갑 주소로 제출 받음
- OCR + 누락 검증 + 신청 자격 판정 자동화

2. **2단계: 신청서 자동완성**
- 프로그램별 양식을 내부 공통 스키마로 정규화
- 사용자 한 번 입력한 정보를 여러 양식으로 변환
- 최종 제출 직전 미리보기와 체크리스트 제공

3. **3단계: 공식 연계 준비**
- 법인 설립
- 전자문서지갑 `API 이용기관` 가능성 검토
- 학교/재단/지자체 같은 권한기관과 파일럿 제휴
- 여기서부터 비로소 “문서 직수신” 고도화를 노릴 수 있음

한 줄로 요약하면, **내일 시작할 MVP는 “발급 자동화”가 아니라 “발급 이후 자동화”**입니다.

1. **토스는 공공 서류를 어떻게 가져오나**
토스가 주민등록등본 같은 서류를 가져오는 방식은, 공개자료상 **정부24/전자문서지갑 체계에 민간앱으로 연계**되는 구조로 보는 게 맞습니다.  
직접 등본 원장 DB를 조회하는 API를 토스가 따로 갖는다는 공개 근거는 못 찾았습니다.

기술적으로는 대체로 이렇게 보입니다.
1. 사용자가 토스 앱에서 `토스인증서` 또는 `모바일 신분증`으로 본인 확인
2. 정부24/전자증명서 발급 시스템이 원본 전자증명서를 생성
3. 그 문서를 `정부 전자문서지갑` 인프라에 저장/유통
4. 토스는 그 지갑과 연계된 **민간 프론트엔드/수신채널** 역할을 수행
5. 제출 시에는 전자문서지갑 주소/QR/기관제출 방식으로 송신

이 해석의 근거:
- 정부24 연말정산 창구가 **2026년 1월 12일**부터 토스 등 민간 앱의 모바일신분증 로그인을 지원한다고 보도됨
- 전자문서지갑 FAQ는 **2026년 2월 기준 59개 공공·민간 앱**에서 전자증명서 이용 가능하다고 밝힘
- 카카오 사례에서는 “카카오 인증서로 본인 인증 후, 정부24 회원정보와 연동해 공문서 신청·발급” 구조가 공개됨. 토스도 같은 계열 구조일 가능성이 높습니다

즉, **토스가 정부24를 대체하는 게 아니라, 정부24/전자문서지갑의 민간 채널이 된 것**에 가깝습니다.

2. **삼쩜삼은 홈택스 데이터를 어떻게 스크래핑하나**
공개자료를 종합하면, 삼쩜삼의 구조는 시기/서비스 유형에 따라 **두 가지가 섞여 있었던 것처럼 보입니다.**

첫째, **사용자 본인 신고형**
- 택스워치 보도에 따르면, 사용자는 삼쩜삼 앱에서 진행하는 것처럼 보이지만 실제론 **홈택스 인증을 거쳐 홈택스에서 본인 명의로 신고**
- 이 경우 삼쩜삼은 `질문지 + 데이터 가공 + 신고 로직 + 브라우저 자동화/세션 처리` 레이어에 가깝습니다

둘째, **연계 세무대리인형**
- 다른 보도에서는 사용자가 입력한 정보를 `연계 세무대리인`에게 넘기고, 그 세무대리인이 홈택스에서 `수임동의 후` 환급액 열람·신청을 대리한다고 설명합니다

기술적으로 추정하면:
1. 사용자 간편인증/홈택스 인증
2. 홈택스 세션 또는 세무대리 권한 확보
3. 원천징수영수증, 지급명세서, 소득자료 등 조회
4. 자체 규칙엔진으로 공제/경비/환급 계산
5. 홈택스 신고화면 자동 입력 또는 대리신고 제출

중요한 점은, **삼쩜삼의 핵심 경쟁력은 “공공 API”가 아니라 “로그인/세션 처리 + 데이터 정규화 + 세법 규칙엔진 + UX”**라는 점입니다.  
반대로 말하면, 이 모델은 계속 **세무대리 책임, 개인정보, 트래픽, 스크래핑 규제 리스크**를 안습니다.

3. **뱅크샐러드의 마이데이터 연동 구조**
이건 비교적 명확합니다. 뱅크샐러드는 `스크래핑`보다 **금융 마이데이터 표준 API** 쪽입니다.

구조는 보통 이렇습니다.
1. 사용자가 뱅크샐러드에서 정보전송 요구
2. 통합인증/동의 절차 수행
3. 각 금융기관이 **표준 API**로 데이터 전송
4. 뱅크샐러드 내부 파이프라인이 데이터를 상품 단위로 정규화

- 뱅크샐러드는 공식 기술 블로그에서 `마이데이터로 들어오는 금융 데이터`를 내부 데이터 파이프라인으로 정리하고, 별도 API 계층으로 제품에 제공한다고 설명
- 고객센터도 “금융기관이 전송하는 데이터를 그대로 보여준다”고 안내

즉 뱅크샐러드는 `원천 수집 = 표준 API`, `차별화 = 데이터 모델링/추천/UX` 구조입니다.

4. **CODEF 없이 공공 데이터를 가져오는 대안**
현실적인 대안은 5가지입니다.

1. **전자문서지갑 수취 구조**
- 사용자가 정부24/발급기관에서 직접 발급
- 귀사 법인 전자문서지갑으로 제출
- 장점: 공동인증서 저장 불필요, 법적 리스크 상대적으로 낮음
- 단점: 사용자가 최초 발급은 직접 해야 함

2. **기관별 공식 API 개별 계약**
- 건강보험, 교육, 자격증, 자동차, 일부 지자체 데이터는 기관별 공개 API 또는 제휴 경로가 존재
- 단점: 주민등록등본 같은 핵심 문서는 대부분 바로 안 열림

3. **민간 앱 연계기관 모델**
- 전자문서지갑 포털은 `API 이용기관`을 위한 절차를 공개하고 있고, `서버인증서 기반 기관·법인`만 표준 API 사용 가능하다고 안내
- 즉, 법인 설립 후 정식 연계는 가능하지만 대학생 팀의 `내일` 플랜은 아님

4. **사용자 업로드 + OCR + 진본검증**
- PDF/전자증명서 업로드
- 문서번호/진본확인 코드/메타데이터 파싱
- 신청서 자동 작성
- 가장 빨리 구현 가능

5. **행정정보공동이용 대상 기관과 제휴**
- 스타트업이 단독으로 쓰기보단, 이미 권한 있는 학교/재단/지자체/금융기관과 붙는 방식
- 본질적으로 `권한 있는 기관의 프론트도어`가 되는 전략입니다

5. **대학생 팀이 공동인증서 없이 사용자 서류를 수집하는 현실적 방법**
가장 현실적인 건 아래 조합입니다.

1. 사용자가 `정부24/홈택스/건보`에서 **간편인증(카카오·PASS·토스 등)** 으로 직접 로그인
2. 서류를 발급받아
3. `업로드`하거나, 가능하면 `전자문서지갑 주소/QR`로 귀사 법인 지갑에 직접 전송
4. 팀은 OCR/파싱/서류 체크/신청서 자동완성만 담당
5. 최종 제출·전자서명은 반드시 사용자가 직접 수행

이 방식이면:
- 공동인증서 보관 불필요
- 사용자 계정 탈취 리스크 감소
- “대리 신청”이 아니라 “신청 도우미”에 더 가까워집니다

6. **정부24 간편인증(카카오/PASS)을 제3자 앱에서 활용할 수 있나**
**정부24의 간편인증 자체를 제3자 앱이 그대로 가져다 쓰는 공개 API는 찾지 못했습니다.**  
대신 두 가지는 가능합니다.

1. **개별 민간전자서명 사업자 연동**
- KISA가 2023년 `민간 이용기관 대상 간편인증 인터페이스 가이드라인`을 발표
- 즉 민간 서비스가 카카오/PASS/토스 같은 인증사업자와 직접 연동하는 건 가능

2. **정부 서비스의 민간채널화**
- 정부24/전자문서지갑/모바일 신분증이 토스·네이버 같은 민간앱으로 개방되는 방식
- 하지만 이건 `정부가 승인한 특정 연계`이지, 아무 앱이나 정부24 로그인 API를 붙일 수 있다는 뜻은 아닙니다

정리하면, **제3자 앱이 정부24의 간편인증 게이트웨이를 재사용하는 건 아니고, 민간 인증사업자를 별도로 붙이거나 정부의 공식 연계 프로그램에 들어가야 합니다.**

7. **최근 정부24가 제3자 연동을 확대한 사례가 있나**
있습니다. 꽤 명확합니다.

- **2026년 1월 12일**: 정부24 연말정산 전용창구가 토스, 네이버, 카카오뱅크, KB스타뱅킹, NH올원뱅크, 삼성월렛 등 **민간 앱 6종 모바일신분증 로그인** 지원
- **2025년**: 모바일 신분증 민간개방이 삼성월렛에서 네이버, 카카오뱅크, 토스, KB, NH 등으로 확대됐다는 보도
- **2026년 2월 기준**: 전자문서지갑은 정부24 외 **59개 공공·민간 앱**에서 이용 가능

즉 정부는 `원천 데이터 API 완전 개방`보다는, **전자문서지갑/모바일신분증/민간앱 채널 확대** 방향으로 움직이고 있습니다.

8. **법적으로 ‘신청 도우미’와 ‘대리 신청’의 경계**
실무적으로는 아래 선이 중요합니다.

`신청 도우미`에 가까운 경우
- 사용자가 직접 정부24/홈택스에 로그인
- 사용자가 직접 최종 제출/서명
- 서비스는 체크리스트, 서류 검증, 자동입력, OCR, 리마인드만 제공
- 서비스가 인증수단·비밀번호·공동인증서를 보관하지 않음

`대리 신청`으로 보기 쉬운 경우
- 서비스가 사용자 계정으로 대신 로그인
- 공동인증서/간편인증 수단을 보관·위임받아 사용
- 서비스가 사용자를 대신해 전자서명
- 적법한 위임장/수임관계 없이 신청을 완료
- 사용자는 “눌렀다고 생각”하지만 실제론 사업자가 법적 행위를 수행

공식 민원 페이지들을 보면, 대리신청이 허용되는 민원도 **위임장 + 본인·대리인 신분확인 서류**를 요구합니다.  
또 `민원 처리에 관한 법률 제10조의2`의 본인정보 공동이용은, 본인이 지정한 **민원처리기관**에 본인정보를 제공하는 제도이지, 임의의 스타트업이 공공DB를 직접 읽는 권한으로 보기 어렵습니다.

9. **대학생 팀이 내일 바로 시작할 현실적 자동화 3단계**
이건 이렇게 가는 게 맞습니다.

1. **1단계: 반자동 수집**
- 사용자에게 정부24/홈택스 발급 링크 제공
- 간편인증으로 직접 발급하게 함
- PDF 업로드 또는 전자문서지갑 주소로 제출 받음
- OCR + 누락 검증 + 신청 자격 판정 자동화

2. **2단계: 신청서 자동완성**
- 프로그램별 양식을 내부 공통 스키마로 정규화
- 사용자 한 번 입력한 정보를 여러 양식으로 변환
- 최종 제출 직전 미리보기와 체크리스트 제공

3. **3단계: 공식 연계 준비**
- 법인 설립
- 전자문서지갑 `API 이용기관` 가능성 검토
- 학교/재단/지자체 같은 권한기관과 파일럿 제휴
- 여기서부터 비로소 “문서 직수신” 고도화를 노릴 수 있음

한 줄로 요약하면, **내일 시작할 MVP는 “발급 자동화”가 아니라 “발급 이후 자동화”**입니다.  
이 경로가 기술적으로도 빠르고, 법적으로도 훨씬 안전합니다.
