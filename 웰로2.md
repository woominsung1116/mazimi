> **목적**: 웰로.md에서 제안한 핵심 기능 3가지의 기술적 실현 가능성 검증

## Part 1: 원스톱 신청 연동

### 최종 판정표

| 검증 항목 | Codex | Gemini | **Claude 합성** |
|-----------|-------|--------|----------------|
| 공공 플랫폼 신청 API | **불가능** | 조건부 가능 | **불가능** (현재) |
| 딥링크 연결 | 조건부 가능 | 조건부 가능 | **조건부 가능** |
| 양식 자동 채우기 | 조건부 가능 | 조건부 가능 | **조건부 가능** |
| 인증/법적 규제 | 조건부 가능 | 조건부 가능 | **조건부 가능** |
| MVP 실현 범위 | 보조형만 | WebView 대응 | **보조형 가능** |

### 상세 분석

#### 1-1. 공공 플랫폼 신청 API → **불가능**
- 정부24/복지로/고용24 모두 **조회/안내용 API만 공개**
- 외부 앱이 프로그래매틱하게 신청서를 제출하는 공개 API는 **존재하지 않음** (Codex 실제 웹 검색 확인)
- 행안부 '디지털서비스 개방(API.digital.go.kr)' 허브가 순차 개방 중이나, 복지 정책 **신청 API는 아직 미개방** (Gemini)
- 토스/카카오는 정부와 **비공개 제휴** 기반으로 WebView 연동. 일반 스타트업은 동일 접근 어려움

#### 1-2. 딥링크 연결 → **조건부 가능**
- 정부24 민원 페이지는 `CappBizCD` 파라미터가 포함된 개별 URL 구조 사용
- 복지로/고용24도 개별 서비스별 신청 경로 존재
- **가능**: 정책 상세 → 공식 신청 페이지 수준의 웹 링크 연결
- **제한**: 로그인/세션/본인인증/서비스 오픈 기간에 따라 중간 화면으로 리다이렉트 가능
- 앱 전용 커스텀 스킴 딥링크는 확인되지 않음

#### 1-3. 양식 자동 채우기 → **조건부 가능**
- **가능**: 앱 내 프로필 데이터로 자체 보조 화면을 미리 채우는 것
- **불가능**: 공식 사이트의 실제 신청 폼을 외부 앱이 안정적으로 자동 채우는 것
- 정부24: 보안 프로그램 + 전자서명 모듈 필요
- 고용24: 추가 본인인증 필요
- 복지로: 공동인증서/간편인증/금융인증서 흐름 사용
- **현실적 방안**: 사전입력 저장 → 공식 사이트 이동 → 사용자가 최종 확인/인증

#### 1-4. 인증/법적 규제 → **조건부 가능**
- 간편인증 SDK(KISA 통합인증창구): 카카오/네이버/PASS 등 12종+ 연동 가능. 기술 난이도 **낮음** (Gemini)
- 전자서명법상 공동인증서 독점 해제됨 — 다양한 인증수단 사용 가능
- **단, 최종 전자서명과 본인확인은 사용자 본인 통제 하에 있어야 함**
- 대리신청: 서비스별 법적 요건이 다름 (복지로는 '부모만 신청 가능' 등 제한)
- 앱 사업자가 사용자를 대신해 서명/신청하는 구조는 **법적 위험 큼**

#### 1-5. 현실적 MVP 범위

| 가능한 것 | 막히는 것 |
|-----------|-----------|
| 정책 추천 + 적합도 점수 | 앱 내 직접 신청 제출 |
| 자격 사전판별 (프로필 기반) | 공식 폼 자동 작성/대리 서명 |
| 서류 체크리스트 자동 생성 | 정부 사이트 양식 자동 채우기 |
| 공식 신청 페이지 정확한 URL 연결 | 범용 원클릭 제출 |
| 앱 내 사전입력 저장 (클립보드 복사) | 사용자 전자서명 대행 |

### 3모델 충돌 해소

## Part 2: 행정데이터(마이데이터) 연동

### 최종 판정표

| 검증 항목 | Codex | Gemini | **Claude 합성** |
|-----------|-------|--------|----------------|
| 공공 마이데이터 API | 조건부 가능 | **구현 가능** | **조건부 가능** |
| 금융 마이데이터 | 사실상 어려움 | 허가 필요 | **MVP 제외** |
| 인허가 절차 | 3개월 심사+실사 | 공익성 심사 | **진입 가능** |
| data.go.kr 범위 | 조건부 가능 | 정적 정보 한정 | **정책 DB용 충분** |
| MVP 데이터 소스 | 가능 | 가능 | **단계별 도입** |

### 상세 분석

#### 2-1. 공공 마이데이터 → **조건부 가능** (이용기관 승인 필수)
- 행안부 '공공 마이데이터 묶음정보 서비스'로 **226종** 행정정보 수신 가능 (Codex)
- 주민등록, 건강보험료, 소득금액증명 등 핵심 데이터 포함
- 민간기관도 '이용기관' 신청 → 심사(~3개월) → 현장실사 → 보안저장소 구축 → API 연계
- 청년 정책 추천 앱은 **공익성 명분이 뚜렷**하여 승인 가능성 높음 (Gemini)
- **핵심 제약**: 아무나 바로 API 키 받아 쓰는 구조가 아님. 보안실사 + 저장소 구축 비용 필요

#### 2-2. 금융 마이데이터 → **MVP 제외 권장**
- 본인신용정보관리업 허가 필요 (금융위원회)
- 자본금 **5억 원 이상**, 보안 설비, 2년 사업계획서 등 제출
- 초기 스타트업에 비현실적 → **기존 허가사업자 제휴**가 유일한 현실적 경로

#### 2-3. data.go.kr 활용 → **정책 카탈로그용 충분**
- 활용 가능: 정부 혜택/공공서비스 메타데이터, 보조금 통계, 고용24 채용/훈련 조회
- 한계: 개인 단위 주민·소득·재산 정보 불가, 일부 API **상업적 이용 금지** 조건 (Codex 확인)
- 결론: **정책 DB 구축에는 유용하나 개인 자격 자동판별은 data.go.kr만으로 불가**

#### 2-4. 스크래핑 vs 공식 API
- **스크래핑: 절대 비권장**
- 개인정보보호법 및 정보통신망법 위반 소지
- 2024년 이후 정부가 강력 지양 권고
- 공공기관 보안 정책 변경 시 수시 차단
- **공식 API만 사용**: 법적 리스크 제로, 데이터 정합성 보장

### 3모델 충돌 해소

| 포인트 | Codex | Gemini | 판정 |
|--------|-------|--------|------|
| 공공 마이데이터 난이도 | "아무나 못 씀" 강조 | "구현 가능" 낙관 | **Codex 쪽이 현실적** — 승인은 가능하나 실사/저장소 비용 부담 |
| 데이터 종 수 | 226종 | 180여 종 | 시점/분류 기준 차이, 실질적 차이 없음 |

### 3-1. 장학금 시장 현황

| 서비스 | 주체 | 특징 | 약점 |
|--------|------|------|------|
| **한국장학재단 앱** | 공공 (KOSAF) | 국가장학금/학자금대출 전문, 서류제출, 근로장학 | 재단 사업 중심, 교내/민간 장학금 미포괄, UX 경직 |
| **드림스폰** | 민간 스타트업 | 민간/기부 장학금 큐레이션 | 업데이트 속도 편차, 필터링 정밀도 부족 |
| **링커리어** | 민간 | 대외활동 + 장학금 통합 | 장학금 전문성 낮음, 커뮤니티 성격 강함 |
| **온통청년** | 고용노동부 | 청년정책 통합 + 자가진단 | 장학금 특화 아님, 포털 성격 |

**시장 공백 (3모델 합의)**:
> 국가장학금 + 교내장학금 + 민간장학금 + 청년정책(주거/생활비/취업)을 **개인별 맞춤 매칭으로 통합 제공하는 서비스가 없다**. 사용자는 여러 앱을 전전하며 자격을 직접 대조해야 함.

### 3-2. 장학금 데이터 소스

| 데이터 소스 | 판정 | 상세 |
|------------|------|------|
| **data.go.kr 학자금지원정보(대학생)** | **가능** | 국가/지자체/민간장학재단/대학 정보 포함. 단, 수기 수집 기반이라 정확성/최신성 한계 명시 |
| **전국대학별장학금정보표준데이터** | 조건부 | 전년도 대학별 수혜액 집계 — 실시간 모집공고용으로는 부족 |
| **온통청년 Open API** | **가능** | 청년정책 통합정보 제공, 장학금 포함 여부는 정책별로 다름 |
| **교내장학금 (대학별)** | **조건부** | 전국 공통 API 없음. 대학별 공지 게시판 크롤링 + LLM 비정형 데이터 추출 필요 (~400개 대학) |
| **민간장학금 (재단별)** | **조건부** | 재단 홈페이지 크롤링 or 드림스폰 등 기존 플랫폼 DB 제휴. 실시간성/정합성은 직접 운영 필요 |
| **KOSAF 자체 개발자 API** | **미확인** | kosaf.go.kr에 공개 개발자용 API 문서는 확인 안 됨. data.go.kr이 실질 연동원 |

### 3-3. 핵심 기술 제약

#### 소득분위(학자금 지원구간) 자동 연동 → **불가능**
- KOSAF는 웹/앱에서 조회·통지서 발급을 제공하지만, **외부 앱이 붙을 수 있는 공개 API 없음** (Codex 웹 검색 확인)
- **대안**:
  - 사용자 직접 입력 (소득구간 선택)
  - 통지서 사진 업로드 → OCR 자동 인식
  - 예상 소득구간 계산기 (가구원 수, 소득, 재산 기반 시뮬레이션)

#### 대학 성적(GPA) 연동 → **조건부 가능**
- 공공 마이데이터 권한 획득 시 일부 가능 (Gemini)
- 대학 포털(GLS, mySNU 등) 크롤링은 보안 강화로 점차 어려워지는 추세
- **현실적 대안**: 사용자 직접 입력 + 성적증명서 업로드(OCR)

#### 장학금 직접 신청 → **불가능**
- 신청 주체가 KOSAF, 대학 포털, 민간재단, 외부 폼으로 **완전 분산**
- 인증/전자서명/개인정보 동의/서류 제출 채널이 제각각
- **현실적 방안**: 딥링크/웹뷰 연결 + 체크리스트 + 진행상태 추적

2. **장학금-정책 패키지 추천**: 학업(장학금) + 생활비(청년수당) + 주거(임차보증금) 한 묶음
3. **합격 가능성 순위**: 과거 수혜자 데이터 기반 "이 장학금에 합격할 확률: 높음/보통/낮음"
4. **학기별 자동 업데이트**: 학년/학기/전공 변경 → 추천 자동 갱신
5. **서류 누락 방지**: 마감일 D-7/D-3/D-1 알림 + 서류 준비 상태 체크리스트

> **Codex 핵심 인사이트**: "그냥 묶는 것만으로는 차별화가 약하다. 이미 온통청년/장학재단이 부분 통합을 하고 있다. 차별화는 **대학생 특화 추천 정확도 + 학기별 여정 UX + 마감/서류 관리**에서 나온다."

## Part 4: 종합 판정

### 전체 기능 판정 매트릭스

| 기능 | 완전형 판정 | 보조/단계형 판정 | MVP 포함 여부 |
|------|-----------|----------------|-------------|
| **원스톱 신청 (API 직접 제출)** | **불가능** | 딥링크+사전입력 가능 | Phase 1 (보조형) |
| **행정데이터 연동 (공공 마이데이터)** | **조건부 가능** | 이용기관 승인 시 가능 | Phase 2 |
| **행정데이터 연동 (금융 마이데이터)** | **비현실적** | 제휴형만 가능 | Phase 3 |
| **장학금 DB 구축** | **가능** | data.go.kr + 크롤링 | Phase 1 |
| **장학금 자격 사전판별** | **조건부 가능** | 룰 기반 1차 필터링 | Phase 1 |
| **장학금 소득분위 자동 연동** | **불가능** | 자가입력/OCR 대안 | Phase 1 (수동) |
| **장학금 원스톱 신청** | **불가능** | 딥링크+체크리스트 | Phase 1 (보조형) |
| **마감일 알림 + 서류 관리** | **가능** | 기술 난이도 낮음 | Phase 1 |

### 단계별 구현 로드맵 (3모델 합의)

```
Phase 1: MVP (0~3개월)
├── 정책 DB: data.go.kr 청년정책 API + 학자금지원정보 API
├── 장학금 DB: KOSAF 공공데이터 + 주요 대학 20개교 크롤링
├── AI 추천: 프로필 기반 맞춤 추천 + 설명 가능한 근거
├── 소득분위: 사용자 직접 입력 or 통지서 OCR
├── 신청 지원: 공식 신청 URL 정확 연결 + 서류 체크리스트
├── 알림: 마감 D-7/D-3/D-1 푸시 + 신규 정책/장학금 알림
└── 수혜 예측: "받을 수 있는 총 금액" 대시보드

Phase 2: 공공 마이데이터 도입 (3~6개월, Phase 1과 병행 신청)
├── 행안부 이용기관 신청 (Phase 1 초기에 즉시 시작)
├── 보안저장소 구축/임대
├── 소득·거주지·가족관계 데이터 API 연동
├── 수동 입력 → 자동 자격 검증 전환
└── 대학 확장: 100개교 → 전국 커버

Phase 3: 생태계 확장 (6~12개월)
├── 디지털서비스 개방 허브 모니터링 → 개방된 신청 API 순차 연동
├── 정부 제휴/협약 추진 (국민비서 등)
├── 금융 마이데이터 사업자 제휴 검토
├── WebView + 간편인증 SDK로 신청 UX 최적화
├── 커뮤니티: 수혜자 후기 + 합격 가능성 데이터 축적
└── 민간장학금 재단 직접 제휴 확대
```

### 핵심 리스크 매트릭스

| 리스크 | 심각도 | 발생 확률 | 대응 전략 |
|--------|--------|----------|-----------|
| 이용기관 승인 거부/지연 | 중 | 낮음 | Phase 1에서 수동 입력만으로도 충분한 가치 제공 |
| data.go.kr 상업적 이용 제한 | 중 | 중 | API별 라이선스 사전 확인, 직접 수집 병행 |
| 장학금 데이터 최신성 부족 | 중 | 높음 | LLM 크롤러 + 변경 감지 파이프라인 구축 |
| 정부 사이트 URL 구조 변경 | 저 | 중 | 주기적 헬스체크 + 자동 알림 |
| 스크래핑 유혹 | 고 | 중 | **절대 금지** — 공식 API만 사용 |
| 대학별 크롤링 차단 | 중 | 중 | 주요 대학 제휴 우선, 크롤링은 공지 게시판 한정 |

## Part 5: 킬러 기능 재정의 (장학금 포함)

웰로.md의 킬러 기능 3개를 장학금 통합 관점에서 업데이트:

### 기존 (웰로.md)
1. AI 챗봇 대화형 추천
2. 수혜금액 시뮬레이터
3. 정책 조합 최적화

### 업데이트 (웰로2.md)
1. **AI 대화형 통합 추천**: "이번 학기 등록금이 부담돼요" → 장학금 + 학자금대출 + 청년수당 한 번에 추천
2. **통합 수혜 대시보드**: 장학금 + 청년정책 = "월 총 XX만원 수혜 가능" 시각화
3. **학기별 정책 매니저**: 학년/전공/소득변화 자동 트래킹 → 놓치는 장학금/정책 0
4. **합격 가능성 AI**: 과거 수혜 데이터 기반 장학금별 합격 확률 예측 (고도화 단계)
5. **서류 완벽 관리**: 정책별+장학금별 필요 서류 통합 체크리스트 + 마감 카운트다운
